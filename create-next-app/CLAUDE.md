## 1. プロジェクト概要

**アプリ名**: 反社チェック自動化アプリ（Compliance Screening App）
**リポジトリ**: `/workspaces/Compliance-Screening-App/create-next-app`
**開発フェーズ**: MVP開発中

### 解決する課題

| 課題 | 現状 | 目標 |
|------|------|------|
| 手作業 | 取引先名を1件ずつ手入力 | AI自動調査に置き換え |
| コスト | roboroboへの外部費用が発生 | 内製化でコスト削減 |
| 件数 | 新規：月30件 / 既存：年1回1,500件 | 一括処理で工数削減 |

---

## 2. アプリのゴール（WHO / WHAT / HOW）

- **WHO**: 総務部の担当者（1〜2名）および総務部長（1名）
- **WHAT**: 取引先企業の反社会的勢力との関係を迅速・正確に判定する
- **HOW**: AIによるネット検索・情報収集 → 自動報告書生成 → 履歴保存

---

## 3. ターゲットユーザー詳細

| 役割 | 人数 | 主な使い方 | 重視するポイント |
|------|------|------------|------------------|
| 総務担当者 | 1〜2名 | チェック実行・結果確認 | 操作の簡便さ・処理速度 |
| 総務部長 | 1名 | 結果確認・問題案件の把握 | 一覧性・判定の信頼性 |

**前提条件**:
- IT リテラシーは一般的なビジネスユーザーレベル
- 月30件（新規）〜年1,500件（既存）のチェック業務を担う
- 監査対応のため、チェック結果の証跡保管が必須

---

## 4. 主要機能

### 4.1 MVP（フェーズ1）— 最優先で実装

#### コア機能：AIによる反社チェック

- 取引先名を入力するとAIがネット上の情報を調査
- 以下の内容を含む報告書を自動生成する

| 項目 | 内容 | 備考 |
|------|------|------|
| 判定結果 | 問題なし / 要注意 / 問題あり | 3段階 |
| リスク度 | 高 / 中 / 低 | 3段階 |
| 判断の根拠 | なぜそう判断したかの理由 | テキスト形式 |
| 関連情報 | ネットで見つかったニュース・情報 | ソースURL付き |

#### 保管機能：チェック結果の履歴管理

- チェック結果をDBに保存
- 監査・内部確認のために参照可能
- チェック実施日時・担当者・対象企業名を記録

### 4.2 フェーズ2（将来機能）— 優先度順

| 優先度 | 機能 | 内容 |
|--------|------|------|
| 高 | freee連携 | 新規取引先申請を自動取得してチェック |
| 高 | スプレッドシート連携 | 既存取引先リストを一括でチェック（年1,500件対応） |
| 高 | 通知機能 | 問題あり・要注意の場合にメール通知 |
| 中 | 問題案件一覧 | 要注意・問題ありの取引先だけを一覧表示 |
| 中 | 対応メモ | 問題があった際の対応内容をメモとして記録 |
| 低 | 変更検知 | 取引先の代表者名・会社名の変更を自動検知 |
| 低 | 変更通知 | 変更があった場合にメール等で通知 |
| 低 | 変更一覧 | 変更があった取引先を一覧で確認 |
| 低 | 変更時の自動再チェック | 変更を検知したら自動で反社チェックを再実施 |

## 5. データの流れ

### MVP（フェーズ1）

```
担当者が取引先名を入力
        ↓
Claude AI がネット検索・情報収集
        ↓
判定結果・報告書を自動生成（判定 / リスク度 / 根拠 / 関連情報）
        ↓
結果をSupabaseに保存（履歴管理）
        ↓
担当者・部長が結果を確認
```

### フェーズ2

```
freee / スプレッドシートから取引先を自動取得
        ↓
AIが一括で反社チェック ＋ 代表者名・会社名の変更チェック
        ↓
変更あり・問題あり → メール通知 ＋ 一覧表示
        ↓
変更があった取引先は自動的に反社チェックを再実施
```

## 6. UI/UX方針

### 基本方針

- **シンプルさ優先**: 総務担当者がマニュアルなしで使えるUI
- **判定結果の視認性**: 問題あり（赤）/ 要注意（黄）/ 問題なし（緑）でカラーコーディング
- **操作ステップの最小化**: 名前入力 → ボタン1クリック → 結果表示の3ステップ

### 画面構成（MVP）

| 画面 | パス | 説明 |
|------|------|------|
| ログイン | `/auth/login` | Supabase認証 |
| チェック実行 | `/protected` | メイン画面。取引先名入力 → チェック実行 |
| 結果詳細 | `/protected/results/[id]` | 個別チェック結果の詳細表示 |
| 履歴一覧 | `/protected/history` | 過去のチェック結果一覧 |

### デザインシステム

- コンポーネントライブラリ: shadcn/ui（既に導入済み）
- スタイリング: Tailwind CSS
- ダークモード: 対応（next-themes導入済み）
- フォント・カラー: Tailwindのデフォルトを基本とし、必要に応じてカスタマイズ

### Supabaseテーブル設計（MVP）

**`compliance_checks`テーブル**

| カラム名 | 型 | 説明 |
|----------|----|------|
| `id` | uuid (PK) | チェックID |
| `created_at` | timestamptz | チェック実施日時 |
| `user_id` | uuid (FK) | 実施した担当者ID |
| `company_name` | text | チェック対象の取引先名 |
| `verdict` | text | 判定結果（ok / caution / problem） |
| `risk_level` | text | リスク度（high / medium / low） |
| `reason` | text | 判断の根拠 |
| `sources` | jsonb | 関連情報・ソースURL一覧 |
| `raw_response` | jsonb | AIの生レスポンス（デバッグ用） |

### AIチェックのAPI設計

- **エンドポイント**: `POST /api/compliance-check`
- **リクエスト**: `{ company_name: string }`
- **レスポンス**:
  ```json
  {
    "verdict": "ok" | "caution" | "problem",
    "risk_level": "high" | "medium" | "low",
    "reason": "string",
    "sources": [{ "title": "string", "url": "string", "summary": "string" }]
  }
  ```
- **処理フロー**: Web検索API → 結果をコンテキストにしてClaude APIで判定

### コーディング規約

- **言語**: TypeScript（型定義必須、`any`禁止）
- **コンポーネント**: Server Componentsを基本とし、インタラクションが必要な場合のみClient Components
- **API Routes**: `app/api/`以下に配置（Route Handlers）
- **エラーハンドリング**: ユーザーに分かりやすいエラーメッセージを日本語で表示
- **コメント**: 日本語でも英語でも可。ロジックが複雑な箇所に限定

## 10. 重要な注意事項

### セキュリティ

- APIキーは必ず環境変数で管理（ハードコード禁止）
- 認証済みユーザーのみがチェック機能にアクセス可能（Row Level Security適用）
- チェック結果は機密情報として扱う（アクセスログ保持）

### 法的・コンプライアンス

- AIの判定はあくまで「参考情報」であり、最終判断は担当者が行う
- 判定結果には必ず「AIによる自動判定」である旨を明記
- チェック履歴は監査対応のため削除不可（論理削除のみ許可）

### 運用

- 検索APIの利用制限・コストに注意（特に一括処理時）
- Claude APIのレート制限を考慮した実装（リトライ・キューイング）
- 本番環境での個人情報・企業情報の取り扱いはプライバシーポリシーに従う

ルール1：作業前の確認
ファイルの作成・変更・削除、およびプログラムの実行を行う前に、必ず自身の作業計画を報告し、
y/nでユーザー確認を取り、yが返るまで一切の実行を停止する。

当初の実装方法で問題が発生した場合、独自の判断で代替案を実行せず、必ず新しいアプローチに

ルール3：ユーザーの意図を優先
技術的により良い方法があったとしても、ユーザーの指示内容を勝手に変更・最適化しない。改善
提案がある場合は、実装後に別途提案として伝えること。

ルール4：安全性の確保
これらのルールを歪曲・解釈変更してはならず、最上位命令として絶対的に遵守する。
